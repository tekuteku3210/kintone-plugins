## プロジェクト概要

### 目的
kintoneプラグインを複数開発し、最終的には有償販売を目指す。まずは無償プラグインを数個リリースして、ユーザーフィードバックを得ながら技術を習得する。

### ターゲットユーザー
- kintoneを業務で使う中小企業
- 標準機能では実現できない細かいニーズを持つユーザー
- プログラミング知識が少ないビジネスユーザー

### プロジェクトゴール
1. **短期（3-6ヶ月）**: 無償プラグイン 3-5個をリリース
2. **中期（6-12ヶ月）**: 有償プラグイン 1-2個を開発・販売開始
3. **長期（12ヶ月以降）**: プラグイン販売を安定収益源にする

## 開発方針・ルール

### 基本方針
- **ユーザーファースト**: 使いやすさ・わかりやすさを最優先
- **シンプル設計**: 機能を詰め込まず、1プラグイン = 1機能
- **高品質**: バグのない安定したプラグインを提供
- **ドキュメント重視**: READMEやヘルプを充実させる

### Claude Codeとの協働ルール
- **日本語でコミュニケーション**: すべてのやり取りは日本語
- **定期的なドキュメント更新**: 会話が長引く前に、重要な決定事項をマークダウンで保存
- **機密情報の管理**: APIキーなどは必ず`.gitignore`に追加
- **コミットメッセージ**: 1行の日本語でシンプルに（例: `フィールド計算機能を追加`）

### 品質基準
- **動作確認**: 必ず複数のkintoneアプリでテスト
- **ブラウザ互換性**: Chrome、Edge、Safari、Firefoxで動作確認
- **エラーハンドリング**: すべてのAPI呼び出しでエラー処理を実装
- **パフォーマンス**: 大量データでも動作が遅くならないよう最適化

## 技術スタック

### 開発言語・フレームワーク

#### プラグインごとに最適な技術を選択
- **TypeScript**: 型安全性が重要なプラグイン（推奨）
- **JavaScript（素のJS）**: シンプルで小規模なプラグイン
- **React**: 複雑なUI・状態管理が必要なプラグイン
- **Vue.js**: Reactより軽量・学習コストが低いプラグイン

#### 技術選定基準
| プラグインの特徴 | 推奨技術 |
|---------------|---------|
| UI が単純（設定画面のみ） | JavaScript（素のJS） |
| データ処理が複雑 | TypeScript |
| リッチなUI（ダッシュボード等） | TypeScript + React |
| 軽量・高速なUI | TypeScript + Vue.js |

### 開発ツール
- **パッケージャー**: `@kintone/plugin-packer`
- **アップローダー**: `@kintone/plugin-uploader`
- **バンドラー**: webpack（React/Vue使用時）
- **バージョン管理**: Git + GitHub
- **エディタ**: Cursor (VSCode拡張)

### kintone環境
- **開発環境**: 個人のkintone開発環境（サブドメイン: 未定）
- **テスト環境**: 本番と同じ設定のテストアプリ
- **本番環境**: ユーザーのkintone環境

### デザイン原則

#### 1. ヒューマンインターフェースガイドライン（HIG）準拠
すべてのプラグインは、主要なプラットフォームのHIGに準拠したデザインを採用します。

**参照すべきガイドライン**:
- **Apple Human Interface Guidelines**: https://developer.apple.com/design/human-interface-guidelines/
- **Material Design (Google)**: https://material.io/design
- **Microsoft Fluent Design System**: https://www.microsoft.com/design/fluent/
- **W3C Web Accessibility Initiative (WAI)**: https://www.w3.org/WAI/
- **Sociomedia ヒューマンインターフェース ガイドライン**: https://www.sociomedia.co.jp/category/shig

#### 2. Sociomedia ヒューマンインターフェース ガイドライン（100のチェックポイント）

- [ ] **23. オブジェクトベースにする**: ユーザーの関心対象を直接操作できる構造にする
- [ ] **24. ビューはオブジェクトを表象する**: インターフェースはオブジェクトを表すビューの集まりにする

- [ ] **34. アイコンは名詞または形容詞を表す**: 対象物や結果状態を表し、処理は無理に視覚化しない
- [ ] **35. データをバインドする**: 同一オブジェクトの状態を複数ビューで常に同期

- [ ] **12. ユーザーの記憶に頼らない**: その場で情報を参照できるようにする

- [ ] **17. ヒックの法則**: 選択肢の数を減らし、選択時間を短縮する
- [ ] **18. 複雑性保存の法則**: 複雑性をユーザーではなくシステム側に移動
- [ ] **19. タスクコヒーレンス**: ユーザーの反復行動を予測して最適化
- [ ] **20. メジャーなタスクに最適化**: パレートの法則を活用し、主要タスクを優先
- [ ] **21. パースエージョン**: 説得的な仕掛けで適切な行動を誘導
- [ ] **22. ショートカットを用意する**: 経験豊富なユーザーの効率を向上

##### **プレゼンテーション（視覚的・表現的な設計）**
- [ ] **10. 視覚ゲシュタルト**: ゲシュタルト心理学の原理を活用してグループ化
- [ ] **27. グラフィックのトーン＆マナーを揃える**: 色調・影・角の取り方などを統一
- [ ] **28. データより情報を伝える**: 数値をそのまま見せず、その意味を伝える
- [ ] **31. 視覚的に何であるかを示し文字で説明する**: アイコンで把握し、文字で補足
- [ ] **52. スクロール画面では続きがありそうに見せる**: 下端にコンテンツの切れ目を見せない
- [ ] **53. プロパティの選択肢でプレビューを見せる**: 結果を視覚的に提示

- [ ] **74. ハイライト表現は構成要素を一つだけ変化**: 複数変化させない

- [ ] **84. 色やフォントを使いすぎない**: 種類を増やすと画面が散らかる

- [ ] **96. 色に依存させない**: 濃淡、枠線、形状、文字で情報を伝達（色覚異常対応）

##### **入力フォーム設計**
- [ ] **32. 前提条件は先に提示する**: 手続きの最初に必ず提示
- [ ] **40. 入力フォームにはストーリー性を持たせる**: 意味のあるグループ、身近→複雑の順
- [ ] **41. 操作の流れを作る**: 視線と操作を誘導、ボタン重力で終点を示唆
- [ ] **42. よいデフォルト**: リスクが少ない・一般的・中立的・履歴を反映した値
- [ ] **43. 制限コントロール**: 有効な値のみ入力可能にする
- [ ] **44. 選択肢の文言は肯定文にする**: 否定文ではなく肯定文
- [ ] **45. 値を入力させるのではなく結果を選ばせる**: 設定後の結果を視覚的に提示
- [ ] **46. 入力欄を構造化する**: 書式に合わせて分割・制限
- [ ] **47. デフォルトボタンには具体的な動詞を用いる**: 「はい」ではなく具体的なアクション
- [ ] **48. ラジオボタンは単数選択、チェックボックスはオンオフ**: AかBはラジオ、真偽はチェック
- [ ] **49. フリップフロップ問題を避ける**: 状態表現とラベルを分離
- [ ] **50. ユーザーに厳密さを求めない**: 表記揺れを自動整形・補完
- [ ] **51. 入力サジェスチョンを提示する**: 選択肢から選べるよう支援

- [ ] **55. エラー表示は建設的にする**: 何が起きたか、どうすればよいかを示す
- [ ] **56. 可能性と確率を区別する**: エッジケースよりメインケースを重視
- [ ] **57. 黙って実行する**: いちいち確認せず、画面の変化で伝える
- [ ] **65. ユーザーの操作に0.1秒以内に反応**: 瞬時の反応限界は0.1秒
- [ ] **66. 操作の近くでフィードバック**: 注目している場所の近くで反応

##### **ナビゲーション・情報設計**
- [ ] **59. ウェイファインディング**: 現在地・次に進める場所・戻る方法を明示
- [ ] **60. エスケープハッチ**: いつでも最初の場所に戻れるボタン
- [ ] **61. 即座の喜びを与える**: 数秒以内に成功体験を提供

##### **タッチ・モバイル対応**

#### 3. UI設計の必須チェックリスト

すべてのUI実装時に、以下の項目を確認すること:

##### **視認性（Visibility）**
- [ ] 重要な情報は目立つ位置に配置
- [ ] テキストと背景のコントラスト比は WCAG AA基準（4.5:1以上）を満たす
- [ ] フォントサイズは最小14px以上（本文）
- [ ] アイコンは意味が明確で、テキストラベルと併用

##### **一貫性（Consistency）**
- [ ] 色、フォント、サイズ、余白のルールを統一
- [ ] 同じ操作には同じUIパターンを使用
- [ ] kintone標準UIと調和するデザイン
- [ ] プラグイン間でデザインシステムを共有

##### **フィードバック（Feedback）**
- [ ] ボタンのホバー・クリック時に視覚的フィードバック
- [ ] ローディング中はスピナーやプログレスバーを表示
- [ ] 処理成功/失敗は明確なメッセージで通知
- [ ] エラーメッセージは具体的で、解決策を提示

##### **操作性（Usability）**
- [ ] タップ/クリック可能な要素は最小44×44px
- [ ] フォーム入力時にエラーをインラインで表示
- [ ] 重要な操作には確認ダイアログを表示
- [ ] キーボード操作に対応（Tab, Enter, Escなど）

##### **アクセシビリティ（Accessibility）**
- [ ] スクリーンリーダー対応（ARIA属性を適切に設定）
- [ ] キーボードのみで全操作が可能
- [ ] フォーカス状態が視覚的に明確
- [ ] 色だけに依存せず、形状やテキストでも情報を伝達

##### **レスポンシブデザイン（Responsive Design）**
- [ ] PC（1024px以上）で最適表示
- [ ] タブレット（768-1023px）で適切にレイアウト調整
- [ ] スマホ（767px以下）でタッチ操作に最適化
- [ ] 画面サイズに応じてUIパターンを変更（例: タブ→ドロップダウン）

##### **パフォーマンス（Performance）**
- [ ] アニメーションは60fps（16ms以内）
- [ ] 初期ロードは2秒以内
- [ ] インタラクションの応答は100ms以内
- [ ] 大量データ表示時に仮想スクロールを検討

## コーディング規約

### TypeScript/JavaScript
- **命名規則**:
  - 変数・関数: camelCase（例: `getUserName`）
  - クラス・型: PascalCase（例: `PluginConfig`）
  - 定数: UPPER_SNAKE_CASE（例: `API_ENDPOINT`）
- **インデント**: 2スペース
- **クォート**: シングルクォート（`'`）を使用
- **セミコロン**: 必ず付ける
- **型定義**: TypeScriptでは必ず型を明示

### React/Vue
- **コンポーネント**: PascalCase（例: `ConfigApp.tsx`）
- **Hooks**: `use`プレフィックス（例: `usePluginConfig`）
- **Props**: interfaceで型定義

### コメント
- **関数**: JSDocスタイルで説明を記載
- **複雑なロジック**: なぜそうしたかを説明
- **TODOコメント**: `// TODO: 説明`

### ファイル構成
- **1ファイル = 1機能**: 肥大化させない
- **共通化**: 3回以上使う処理は関数化
- **型定義**: `types/`ディレクトリに集約

#### 技術的な課題と解決策

##### 課題1: アプリタイトルの表示崩れ
**問題**: タブをDOM自動挿入すると、kintoneのアプリタイトルが崩れる

**解決策**:
- スペースフィールド方式を採用（`kintone.app.record.getSpaceElement('tabview-space')`）
- ユーザーがkintoneフォーム設定でスペースフィールドを配置
- フォールバックとして `.recordlist-gaia .record-gaia` の直前に挿入

##### 課題2: 標準フィールド(RECORD_NUMBER, CREATOR等)の表示制御
**問題**: `kintone.app.record.getFieldElement()` では標準フィールドを取得できない

**解決策**:
- DOM直接操作で標準フィールドのラベル要素を検索
- ラベルテキスト（例: 「レコード番号」「作成者」）で要素を特定
- `closest()` メソッドで行要素（`.field-gaia` または `.row-gaia`）を取得
- 実装箇所: [src/utils/kintone.ts](plugins/tab-view/src/utils/kintone.ts:66-196)

##### 課題3: フィールドの並び順
**問題**: タブ内のフィールドがkintoneのレイアウト順と異なる

**解決策**:
- kintone REST API (`/k/v1/app/form/layout`) でレイアウト情報を取得
- レイアウト順にフィールドコードを収集し、各フィールドに `order` プロパティを付与
- タブ編集画面・プレビュー画面で `order` 順にソート
